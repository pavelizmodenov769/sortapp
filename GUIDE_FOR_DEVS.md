## ⚠️ Главное правило перед стартом

Все четверо зависят от фундамента Dev 1 и от того, что **договорённости на день 1–2 зафиксированы**. Dev 2 не сможет написать сортировки, пока не знает, как выглядит `Student` и `CustomList`. Dev 3 не сможет собирать данные без них же. Dev 4 не сможет собирать меню без всего остального.

Поэтому в **день 1–2** команда **не разбегается по коду**, а вместе (на бумаге/доске) договаривается:

1. Какие поля у `Student`?
2. Какие методы у `CustomList` (`add`, `get`, `set`, `remove`, `size`...)?
3. Какой будет интерфейс `SortStrategy`?
4. Какой формат файла для ввода/вывода?
5. Какие пункты нужны в меню?
(Смотрите CONTRACTS)

## 👤 Dev 1 — Фундамент

**Зона ответственности:** модель `Student`, кастомная коллекция, исключения.
**Ветка:** `feature/domain` — мержится **первой**, от неё зависят все.

### Что делает
Создаёт «кирпичики», из которых остальные строят свои модули. Если Dev 1 ошибётся — пострадает вся команда.

### Шаги

#### 1. Создать структуру проекта
Первым создаёт пакеты:
```
com/team/sortapp/
├── com.team.sortapp.model/        ← сюда Student
├── com.team.sortapp.collection/   ← сюда CustomList
└── exception/    ← сюда ValidationException
```

#### 2. Класс `Student` + паттерн Builder
Сердце проекта. Builder нужен (требование задания), чтобы создание было читаемым:

**Что реализовать внутри:**
- Три поля: `groupNumber` (int), `averageScore` (double), `gradeBookNumber` (long).
- Поля `final` — объект неизменяемый (потокобезопасный, удобен для сортировки).
- Геттеры на каждое поле, **без сеттеров**.
- В `build()` — проверки: группа > 0, балл в [0; 10], зачётка > 0. Нарушение → `ValidationException`.
- Переопределить `equals`, `hashCode`, `toString`.

#### 3. `ValidationException`
Наследник `RuntimeException` с конструктором под сообщение. Единая ошибка для всей команды при невалидных данных.

#### 4. Кастомная коллекция (требование 3★)
Нельзя `ArrayList` — коллекции должны быть кастомными.
- Интерфейс `CustomList<T>`: `add`, `get`, `set`, `remove`, `size`, `isEmpty`, `clear`.
- Реализация `CustomArrayList<T>` — внутри `Object[]`, который **сам растёт** при заполнении (×1.5).
- Свой итератор (для `for-each` и Stream API — нужно Dev 3).
- Проверка индексов → `IndexOutOfBoundsException`.

#### 5. Тесты
- Builder с валидными данными → создаётся.
- Builder с невалидными (балл −1, группа 0) → исключение.
- Коллекция: 1000 `add` (рост), `remove` со сдвигом, неверный индекс → исключение.

### Ключевая ответственность
Первым мержит ветку в `master`. Пока он не отдаст `Student` и `CustomList`, команда работает «вслепую». Задерживаться нельзя.

## 👤 Dev 2 — Сортировки

**Зона ответственности:** алгоритмы сортировки, компараторы, доп. задание 1 «чёт/нечёт».
**Ветка:** `feature/sorting`.

> Dev 2 не начинает, пока Dev 1 не зафиксирует `Student` и `CustomList`.
### Шаги

#### 1. Интерфейс `SortStrategy` (паттерн «Стратегия»)

#### 2. `SortContext`
Обёртка, хранящая текущую стратегию и делегирующая ей работу. Нужна, чтобы UI менял алгоритм «на лету» через `setStrategy(...)`.

#### 3. ДВА алгоритма сортировки ВРУЧНУЮ
**Строгое ограничение:** нельзя `Arrays.sort`, `Collections.sort`, `list.sort()`.
- **`BubbleSort`** — простой, стабильный: два вложенных цикла, сравнение соседей, обмен.
- **`QuickSort`** — рекурсивный: опорный элемент, разделение, рекурсия по частям.

Сортировка идёт **по компаратору**, передаваемому параметром — так один алгоритм сортирует по любому полю.

#### 4. Три компаратора (по полям)
Каждый тривиальный, одна строка:
- `StudentGroupComparator` → `Integer.compare` по группе.
- `StudentScoreComparator` → `Double.compare` по баллу.
- `StudentGradeBookComparator` → `Long.compare` по зачётке.

#### 5. Доп. задание 1 — «чёт/нечёт»
По выбранному полю (например, группа) **чётные** сортируются в натуральном порядке, **нечётные остаются на месте**.

**Алгоритм:**
1. Запомнить индексы чётных элементов.
2. Собрать их значения.
3. Отсортировать значения (своей сортировкой из шага 3).
4. Расставить обратно по тем же индексам.

**Пример для проверки:**
```
исходно:   [3, 2, 5, 8, 4, 7]
чётные на индексах 1,3,4 → [2,8,4] → сортируем → [2,4,8]
результат: [3, 2, 5, 4, 8, 7]   ← нечётные 3,5,7 не сдвинулись
```

#### 6. Тесты
- Каждый алгоритм на: пустой список, 1 элемент, отсортированный, обратно-отсортированный, с дубликатами.
- Проверка, что **нечётные остались на местах** (любят проверять на защите).

### Ключевая ответственность
Самый большой объём. Но компараторы — минуты, а «чёт/нечёт» переиспользует уже написанную сортировку. Главное — запрет готовых сортировок.

## 👤 Dev 3 — Ввод, вывод, валидация, стримы

**Зона ответственности:** взятие данных снаружи (файл / клавиатура / генератор), проверка, возврат результата в файл.
**Ветка:** `feature/io` (мержится параллельно с `feature/sorting`).

### Чем отличается
Больше всего точек соприкосновения с пользователем. Его код — то, что человек реально видит и трогает. Кривая валидация = постоянные падения. Зато самая «осязаемая» часть: ввёл → проверилось → попало в коллекцию.

> Пока Dev 1 не отдал код, Dev 3 работает на «болванках» — заглушках, ведущих себя как `Student`, но без бизнес-логики. Парсер файлов и UI пишутся против интерфейсов уже на день 3.

### Шаги

#### 1. Валидатор `StudentValidator` (ОСНОВА всего)
Первое, за что берётся — все три источника данных будут его использовать. Правила в одном месте, без дублирования.
- Принимает «сырые» строки.
- Парсит в число (`Integer.parseInt`, `Double.parseDouble`, `Long.parseLong`).
- Проверяет диапазоны: группа > 0, балл [0; 10], зачётка > 0.
- Нарушение → `ValidationException` с понятным текстом: «Средний балл должен быть от 0 до 10, получено: 15.7».

**Ловушка новичков:** забывают перехватить `NumberFormatException`. Ввод `"привет"` вместо числа → `parseInt` падает сам с непонятной ошибкой. Валидатор обязан **перехватывать** и оборачивать в свой `ValidationException`.

#### 2. Три источника данных (интерфейс `DataProvider`)
Один метод `provide(int count)` → `CustomList<Student>`. Три реализации:

**А. `RandomDataProvider`** — самый простой:
- Генерирует `count` студентов через `Random` в валидных диапазонах.
- Даже случайные данные должны проходить валидацию (гарантия от мусора).

**Б. `ManualDataProvider`** — с клавиатуры через `Scanner`:
- Запрашивает три строки на студента.
- Каждое поле через валидатор.
- **Ключевое:** при ошибке — перезапрос **именно этого поля**, а не всего студента.
- Подсказки: «Введите номер группы:».

**В. `FileDataProvider`** — построчное чтение:
- Формат строки: `groupNumber,averageScore,gradeBookNumber` (через запятую).
- Сплит по запятой, трим, парсинг через валидатор.
- **Решение команды:** кривую строку **пропускать с предупреждением** («Строка 5 некорректна, пропущена»), не роняя загрузку.

#### 3. Заполнение через стримы (доп. задание 3)
**Подвох:** `CustomList` не реализует `java.util.Collection`, стандартный `Collectors.toList()` не сработает. Аккумулировать вручную.

**Как:** утилита `CollectionFiller` принимает Stream и наполняет `CustomList` через `stream.forEach(list::add)`. Требование формально закрыто.
- `RandomDataProvider` — `Stream.generate(...).limit(count)`.
- `FileDataProvider` — `Files.lines(path).map(line -> parse(line))`.

#### 4. Запись в файл в режиме добавления (доп. задание 2)
`ResultWriter.appendToFile(...)`:
- Открывает файл в режиме **APPEND** (дописать, не перезаписать).
- Метка времени в начале блока.
- По строке на студента.
- Формат согласован с `FileDataProvider` (чтобы читать обратно).

**Что забывают:** проверить, что повторная запись **не затирает** старое. Отдельный тест: записать дважды → два блока.

#### 5. Тесты
- Валидатор: все варианты мусора → корректное исключение.
- `FileDataProvider`: нормальный файл, с кривой строкой, пустой.
- `ResultWriter`: две записи подряд → два блока, старое на месте.
- `CollectionFiller`: ровно `count` студентов.

### Ключевая ответственность
Его код видит пользователь напрямую. Падения на кривом вводе — главный риск.

## 👤 Dev 4 — Многопоточность, UI, сборка (интегратор)

**Зона ответственности:** меню, главный цикл, многопоточная функция, точка входа, Git/сборка.
**Ветка:** `feature/threading-ui` — мержится **последней**.

### Чем отличается
Последний в цепочке. Его ветка мержится последней, потому что в ней сходится всё. Чужие баги всплывают именно у него на интеграции.
### Шаги

#### 1. Многопоточный счётчик (доп. задание 4)
`OccurrenceCounter` — считает вхождения элемента N в несколько потоков.

**Алгоритм:**
1. Число ядер: `Runtime.getRuntime().availableProcessors()`.
2. Разбить коллекцию на чанки — по одному на поток.
3. Каждый поток считает вхождения **в своём куске**.
4. Главный ждёт все (`Future.get()` / `Thread.join()`) и **суммирует**.
5. Печать: «Значение N встречается X раз».

**Две ловушки:**
- **Нельзя** `Stream.parallel()` — готовая реализация, снимают баллы. Только `Thread`/`ExecutorService` руками.
- **Гонка данных:** потоки не пишут в общий счётчик одновременно. Каждый считает локально, сумму собирает главный.

**Что считать «элементом N»:** конкретное значение числового поля (например, номер зачётки). Уточнить с командой заранее.

#### 2. Главное меню и цикл (`Menu`)
Скелет программы. Цикл `while (running)`: печать пунктов → чтение выбора → действие → возврат в начало.

**Пункты меню (примерно):**
1. Способ заполнения (файл / рандом / вручную) + длину.
2. Алгоритм сортировки (пузырёк / быстрая).
3. Поле сортировки (группа / балл / зачётка) **ИЛИ** доп. «чёт/нечёт».
4. Выполнить сортировку и показать результат.
5. Записать результат в файл (`ResultWriter` от Dev 3).
6. Подсчитать вхождения элемента N (многопоточно).
7. **Выход.**

**Ключевое:** выход **только** через пункт «Выход». Никаких `System.exit` посередине.

**Защита от дурака:** буквы вместо числа → не падать, перезапросить.

#### 3. Точка входа `Application`
Класс с `main`. Сборка всех компонентов **вручную** (без DI-фреймворков):
- Создаёт источники данных, стратегии, компараторы.
- Создаёт `SortContext` (от Dev 2) и `Menu`.
- Передаёт в `Menu`, запускает `menu.run()`.

#### 4. Git, README, сборка (как лид-интегратор)
- Репозиторий на GitHub/GitLab создан.
- Ветки `feature/...` соответствуют 4 разработчикам.
- Порядок слияния: `domain` → (`sorting` ‖ `io`) → `threading-ui` → `master`.
- `README` с инструкциями по сборке и запуску.
- `.gitignore` не пускает мусор (`.class`, `target/`, `.idea/`).

#### 5. Тесты
- `OccurrenceCounter`: с дубликатами → корректный счёт; нет элемента → 0; **сравнить** однопоточный и многопоточный результат на 100 000 элементов → совпадают.
- Логику меню (разбор номера пункта, неверный ввод) вынести в метод и тестировать.

### Ключевая ответственность
«Клей» проекта. Главная опасность — не свои баги, а **чужие**, всплывающие на интеграции. Начать раньше и на заглушках.

| | Dev 1 | Dev 2 | Dev 3 | Dev 4 |
|---|---|---|---|---|
| **Что создаёт** | Студент, коллекция, исключения           | Алгоритмы, компараторы, «чёт/нечёт» | Ввод/вывод, валидация, стримы, writer | Многопоточность, меню, сборка |

### Порядок слияния веток
```
feature/domain ─► feature/sorting ─┐
                                    ├──► feature/threading-ui ──► master
feature/io ─────────────────────────┘
```

### Суть ролей одной строкой
- **Dev 1** даёт «гайки и болты».
- **Dev 2** собирает из них «движок сортировки».
- **Dev 3** добывает и проверяет данные.
- **Dev 4** собирает всё в работающую программу и подаёт пользователю.

Без Dev 4 программа — набор классов; без Dev 3 — программе нечего сортировать.
