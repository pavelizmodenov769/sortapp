## 📐 Принципы работы по контрактам

1. Контракты зафиксированы раньше кода.
2. Каждый кодит против интерфейсов, а не против реализаций. Это позволяет работать параллельно: Dev 2/3/4 могут использовать `Student`/`CustomList`, даже если Dev 1 ещё не отдал их финальные версии.
3. Контракт нельзя менять молча. Любое изменение — обсуждение с командой + коммит в `master` с обновлением этого файла.

## Контракт 1. Поля класса `Student`

**Владелец:** Dev 1 (`feature/domain`).

| Поле             | Тип         | Диапазон / инвариант| Обоснование |
| `groupNumber`    | **`int`**   | `> 0`               | Номер группы — небольшое число (1–999). `int` достаточен, `long` избыточен. |
| `averageScore`   | **`double`**| `[0.0; 10.0]`       | Балл бывает дробным (8.5). `double` точнее `float`; диапазон 0–10 — стандартная 10-балльная шкала. |
| `gradeBookNumber`| **`long`**  | `> 0`               | Зачётка — длинное число (напр. `20230001`, 8+ цифр). `int` переполнится при ~2.1 млрд — небезопасно. |

**Дополнительные решения:**
- Все поля **`final`** → объект неизменяемый (immutable). Это делает его потокобезопасным (важно для Dev 4) и безопасным для сортировки.
- Только **геттеры**, без сеттеров.
- Для доп. задания 1 (чёт/нечёт) все три поля числовые — чёт/нечёт можно проверять по любому; **по умолчанию берётся `groupNumber`**.
- Переопределить `equals`, `hashCode`, `toString`.
- Создание — только через **Builder** (паттерн, ручная реализация без Lombok).
- Проверка инвариантов в `Builder.build()` → нарушение кидает `ValidationException`.

## Контракт 2. Интерфейс `CustomList<T>`

**Владелец:** Dev 1 (`feature/domain`).

| Метод                        | Зачем               | Кому критично |
| `void add(T element)`        | Добавить в конец    | Dev 3 (заполнение) |
| `T get(int index)`           | Прочитать элемент   | Dev 2 (сравнение при сортировке) |
| `T set(int index, T element)`| **Заменить** элемент| Dev 2 (обмен при сортировке — без этого сортировать in-place невозможно) |
| `int size()`                 | Длина коллекции     | Dev 2 (границы циклов), Dev 4 |
| `boolean isEmpty()`          | Проверка пустоты    | Все (edge-cases) |
| `T remove(int index)`        | Удалить со сдвигом  | UI / общий API |
| `void clear()` | Сбросить    | UI (повторное заполнение) |
| `void forEach(Consumer)`     | Перебор             | Dev 3 (стримы), вывод |
| `Iterator<T> iterator()`     | Итерация            | **Обязательно** `extends Iterable<T>` — без этого не работают `for-each` и Stream API |

**Ключевые решения:**
- `CustomList<T> extends Iterable<T>` — **критично**, иначе Dev 3 не сможет использовать стримы.
- `set` — **обязателен**. Сортировка in-place требует обмена значений: `list.set(i, list.get(j))`.
- Проверка индексов во всех методах доступа → `IndexOutOfBoundsException`.
- Коллекция реализована на динамическом массиве (`CustomArrayList`), растущем ×1.5 при заполнении.
- **Не потокобезопасна** — для конкурентного доступа Dev 4 использует внешний механизм (чанки).

## Контракт 3. Интерфейс `SortStrategy<T>`

**Владелец:** Dev 2 (`feature/sorting`).

| Элемент решения                       | Выбор                                 | Обоснование |
| `sort(CustomList<T>, Comparator)`     | Сортировка **in-place** по компаратору| Возвращает `void` — меняет коллекцию на месте. Strategy можно применить несколько раз без копий. |
| `Comparator<? super T>`               | Wildcard `? super T`                  | PECS-принцип: компаратор умеет сравнивать `T` и его предков. Стандартная практика Java. |
| `Comparator` от `java.util.Comparator`| **Разрешено**                         | ТЗ: «для кастомной сортировки разрешено использовать компаратор». Запрещены только **готовые сортировки** (`Arrays.sort` и т.д.), не сам интерфейс компаратора. |
| `getName()` | Возвращает читаемое имя | Нужно Dev 4 для меню: «1 — Сортировка пузырьком, 2 — Быстрая». |

**Интерпретация паттерна Strategy (суть для защиты):**
- **Strategy** = *алгоритм* (BubbleSort, QuickSort — отдельные классы-реализации).
- **Comparator** = *правило сравнения по полю* (3 компаратора от Dev 2).
- Комбинаторика: `2 стратегии × 3 компаратора = 6 комбинаций` сортировки «из коробки», плюс «чёт/нечёт» как отдельная стратегия.

**Запрещено:**
- `Arrays.sort`, `Collections.sort`, `List.sort`, `Stream.sorted()`.

## Контракт 4. Формат файла (вход ≡ выход)

**Владелец:** Dev 3 (`feature/io`): `FileDataProvider` (чтение), `ResultWriter` (запись).

> Критично: формат **входа и выхода одинаковый**. Иначе результат сортировки нельзя прочитать обратно — круг замкнут не будет.

**Решение: CSV, по одной строке на студента.**

```csv
101,8.5,20230001
102,7.2,20230002
103,9.0,20230003
```

| Элемент решения       | Выбор                                   | Обоснование |
| **Разделитель полей** | запятая `,`                             | Проще парсить (`split(",")`), читать глазами, править в Блокноте. |
| **Разделитель дроби** | **точка** `.` (не запятая!)             | Если дробь через запятую (`8,5`), она сольётся с разделителем полей → 4 «поля» вместо 3. Плюс `Double.parseDouble("8.5")` локаль-независима, а через запятую упадёт на русской локали. |
| **Заголовок**         | **без заголовка**                       | Минимализм для новичков: первая строка — уже данные. |
| **Комментарии**       | строки, начинающиеся с `#`, игнорируются| Даёт возможность оставлять пометки в файле без поломки парсера. |
| **Пустые строки**     | пропускать                              | Файл с лишним переносом в конце не должен ломаться. |
| **Пробелы**           | `trim` каждого поля                     | `101, 8.5 , 20230001` работает как `101,8.5,20230001`. |
| **Кодировка**         | **UTF-8** обязательно                   | Важно для `ResultWriter` и русского в комментариях. |
| **Невалидная строка** | пропускать с предупреждением в консоль  | «Строка 5 некорректна, пропущена» — не ронять всю загрузку ради одной опечатки. |
| **Расширение**        | `.csv` (`.txt` тоже принимать)          | Стандартное расширение для такого формата. |

**Расположение полей строго фиксировано (порядок важен!):**
```
[0] groupNumber     → int,    > 0
[1] averageScore    → double, [0.0; 10.0]
[2] gradeBookNumber → long,   > 0
```

**Для `ResultWriter` (выход):** перед блоком данных — служебная строка-метка времени:
```csv
# === Отсортировано: 2026-06-28 14:30:05 ===
101,8.5,20230001
102,7.2,20230002
```
Строка начинается с `#`, поэтому при обратном чтении она игнорируется — **round-trip работает**.

## Контракт 5. Интерфейс `DataProvider`

**Владелец:** Dev 3 (`feature/io`).


| Элемент решения               | Выбор                                    | Обоснование |
| **Генерик `DataProvider<T>`?**| **Нет** — конкретно под `Student`        | Проект сортирует только студентов. Обобщение `T` добавит сложности (новички не изучали) без выгоды. |
| **Возвращаемое**              | `CustomList<Student>`, **никогда `null`**| Никаких null-проверок у вызывающего. Пустой результат → пустая коллекция. |
| **Параметр `count`**          | «целевой размер»                         | Для Random/Manual → ровно `count`. Для File → `count` как **верхний лимит** (реально может быть меньше). Эта асимметрия документируется явно. |
| **`count < 0`**               | `IllegalArgumentException`               | Отрицательный размер бессмысленен. `count == 0` → пустая коллекция. |
| **`getName()`**               | читаемое имя для меню                    | Dev 4 выводит: «1 — Случайные данные», «2 — Ввод вручную», «3 — Из файла». |
| **Зависимости (Scanner/Path)**| в **конструкторе**, не в методе          | Интерфейс остаётся чистым. `ManualDataProvider(Scanner)`, `FileDataProvider(Path)`, `RandomDataProvider()`. |
| **Обработка ошибок**          | `ValidationException`                    | Для Manual — перезапрос *внутри* `provide`; для File — skip-and-warn; для Random — генерация заведомо валидных значений. |

## 🔗 Сквозной пример (все 5 контрактов вместе)

```java
// Контракт 5: Dev 4 (меню) единообразно работает с любым источником
DataProvider source = new RandomDataProvider();          // или new FileDataProvider(Path.of("students.csv"))
CustomList<Student> students = source.provide(100);      // Контракт 5 + 2: вернуть CustomList<Student>, никогда null

// Контракт 1: Student создан через Builder, поля final, инварианты проверены
// Контракт 2: CustomList итерируем (Iterable), есть get/set/add

// Контракт 3: Dev 2 дал стратегию + компаратор, сортировка in-place
SortContext<Student> ctx = new SortContext<>(new BubbleSort<>());
ctx.execute(students, new StudentGroupComparator());     // сортирует по группе, на месте

// Контракт 4: Dev 3 записывает результат тем же форматом, что и вход
ResultWriter.appendToFile(Path.of("result.csv"), students);  // APPEND, старое не затрётся
```

```csv
# Файл students.csv — читается FileDataProvider'ом и пишется ResultWriter'ом одним форматом:
101,8.5,20230001
102,7.2,20230002
```
